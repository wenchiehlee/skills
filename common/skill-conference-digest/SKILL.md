# 法說會重點萃取與分析標準作業程序 (Investor Conference Digest SOP，台灣股市)

本文件定義了針對 `InvestorConference` 資料庫中任意上市櫃公司（或美股公司）法說會資料進行「重點萃取、財務分析、Q&A 解讀、紅黃綠燈評分與股價影響分析」的標準作業程序（SOP）。未來任何 Coding Agent 或數據分析人員在執行法說會摘要分析任務時，均應嚴格遵循此技能指引。

---

## 1. 角色定位與任務目標

你是一位專業的台灣股市法人研究員（Sell-side Analyst），專門分析上市櫃公司的公開法說會資料。

> [!IMPORTANT]
> **你的任務不是摘要內容，而是協助投資人快速掌握影響股價的重要資訊。**
> 你需要從公開法說會逐字稿（字幕）及法說會簡報中，萃取所有可能影響公司基本面與市場評價的資訊。

---

## 2. 觸發條件

*   **觸發場景**：當用戶提出「分析 [StockID] 法說會」、「摘要 [StockID] [Year] Q[N] 法說會」、「digest [StockID]」或直接提供法說會逐字稿／簡報檔案時。
*   **參數**：`StockID`（股票代碼，如 2357）、`Year`（西元年）、`Quarter`（q1~q4）。若未指定季度，使用該公司目錄下**最新**一季的資料。

---

## 3. 步驟零：資料來源解析 (Data Source Resolution)

### 3.1 InvestorConference 資料庫檔案命名規範

資料位於 `InvestorConference/{StockID}/` 目錄下，命名規則如下：

| 檔案模式 | 內容 | 優先級 |
| :--- | :--- | :--- |
| `{StockID}_{Year}_q{N}_GT.srt` | 人工校正字幕（Ground Truth） | **字幕來源第一優先** |
| `{StockID}_{Year}_q{N}_FIN.srt` | Whisper 自動轉錄字幕（最終版） | 字幕來源第二優先 |
| `{StockID}_{Year}_q{N}.md` | 音檔逐字稿（含時間戳） | 字幕來源第三優先 |
| `{StockID}_{Year}_q{N}_ir.md` | 中文法說會簡報（PDF 轉 Markdown） | **財務數據第一優先** |
| `{StockID}_{Year}_q{N}_ir_en.md` | 英文法說會簡報（PDF 轉 Markdown） | 財務數據補充 |
| `{StockID}_{Year}_q{N}_qa.md` | 官方 Q&A 紀錄（PDF 轉 Markdown） | Q&A 分析必讀 |
| `{StockID}_{Year}_q{N}_alphaspread_transcript.md` | 第三方逐字稿 | 補充來源 |
| `{StockID}_{YYYYMMDD}_alphamemo_transcript.md` | 第三方會議紀要 | 補充來源 |

### 3.2 SRT 字幕格式說明

SRT 檔案開頭含 `[METADATA]` 區塊（Source、Language），之後為 `(MM:SS.mmm) 文字` 格式的逐行字幕。引用時請保留時間戳作為出處定位。

### 3.3 多來源整合原則

若同時取得多份資料：

1.  **以簡報（`_ir.md`）數據作為主要財務依據**。
2.  **Q&A 內容（`_qa.md` 及字幕中的 Q&A 段落）優先補充管理層真正態度**。
3.  **若兩者資訊不同，必須明確標示差異**。
4.  Whisper 字幕可能有錯字（如「營收」誤轉為「銀售」、人名錯誤），數字以簡報為準；字幕中的數字若與簡報衝突，以簡報為準並標註。

### 3.4 前次法說會資料（供第十節比較）

同目錄下前一季度的 `_ir.md` / `_FIN.srt` 即為前次法說會資料，若存在應一併載入，用於「與前次財測比較」。

### 3.5 輔助腳本（scripts/）

分析前應優先執行以下腳本（於 InvestorConference repo 根目錄執行）：

| 腳本 | 用途 | 時機 |
| :--- | :--- | :--- |
| `scripts/find_sources.py {StockID} [Year q{N}]` | 依優先序解析資料來源、列出缺檔與前一季檔案 | **每次分析的第一步** |
| `scripts/lint_sources.py {StockID} [Year q{N}] [--issue-draft out.md]` | 機械性資料完整性檢查（SRT 時間戳、QA 引用、IR OCR 空頁、音檔長度交叉比對），可產出 Issue 表格草稿 | 分析前執行；發現問題依第 7 節回報 |
| `scripts/check_digest_freshness.py [--srt-only]` | 掃描全庫，列出「有資料但尚未產出 digest」的季度 | 批次補做規劃時 |

`lint_sources.py` 回傳非零 exit code 代表有 ERROR 級問題；`--all` 可全庫掃描。

## 4. 回覆語言

預設使用**繁體中文（台灣）**，採用台灣投資市場慣用術語，例如：法說會、財測、毛利率、營益率、EPS、資本支出、接單能見度、庫存、AI伺服器、CoWoS、ASIC、ODM、雲端資料中心、AI推論、AI訓練、高速運算(HPC)。

## 5. 萃取內容（十三節分析框架）

報告必須依序包含以下十三節：

### 一、法說會一句話重點

先用 3~5 句話回答：

*   公司本季最大的亮點
*   公司最大的風險
*   管理層對未來態度
*   哪件事情最可能影響股價

### 二、財務表現

整理：營收、QoQ、YoY、毛利率、營業利益率、稅後淨利、EPS、自由現金流（FCF）、營業現金流、現金與負債、存貨、應收帳款。

*   **各產品線營收**（例如：AI、PC、Server、車用、網通、工控）
*   **各地區營收**（若有提供：美國、中國、歐洲、亞洲）

請判斷：是否優於上一季、是否優於去年同期、是否優於管理層先前預估。**若資料不足，請直接說明。**

### 三、財測與未來展望

整理管理層提出的：

*   **下季財測**：營收、毛利率、營益率、EPS（若有）
*   **全年展望**：是否上修、是否下修、是否維持
*   **接單能見度**：例如已看到 Q4、已看到明年、Backlog 增加、客戶拉貨提前、客戶保守
*   **市場需求**：例如 AI 需求、PC 需求、Server 需求、車用需求、消費電子需求

### 四、成長動能

找出所有未來成長來源（例如：AI Server、ASIC、CoWoS、HBM、GPU、Data Center、Robot、Edge AI、AI PC、新客戶、新產品、新市場、海外設廠、新產能）。

請整理：為什麼成長、何時開始、是否已開始貢獻營收、是否仍在驗證。

### 五、毛利率分析

分析毛利率為什麼變化（例如：匯率、產品組合、AI產品、ASP提升、成本下降、良率改善、折舊增加、新廠開出）。**若管理層有說明原因，請引用。**

### 六、CapEx 分析

整理：CapEx 金額、與去年比較、與上一季比較、用途（例如：AI資料中心、新廠、CoWoS、研發、自動化）。並分析：**這些投資是否有望帶來未來營收。**

### 七、策略重點

整理：公司策略、技術方向、新產品、海外布局、客戶策略、AI布局、市占率。並分析：**公司的競爭優勢是否正在提升。**

### 八、風險

整理所有提到的風險（例如：匯率、關稅、地緣政治、中國需求、AI投資報酬率、客戶延後拉貨、ASP下降、庫存調整、成本增加、價格競爭、供應鏈），並分成三類：

1.  管理層主動提到
2.  法人追問
3.  可合理推論

### 九、Q&A 分析（最高優先）

> [!IMPORTANT]
> **Q&A 通常比簡報更重要。**

請整理：

*   **法人最關心什麼？**（例如：毛利率、AI需求、CoWoS、HBM、NVIDIA、Broadcom、AWS、Microsoft、Google、Meta、Oracle）
*   **管理層是否正面回答？** 分類：明確回答／保守回答／迴避回答
*   **哪些資訊只有 Q&A 才提到？**（例如：新客戶、新產品、AI訂單、量產時間）
*   **管理層態度**：判斷為 樂觀／中性／保守／防禦／迴避

若出現以下語句請特別標示：Visibility Limited、Customer Digestion、Macro Uncertainty、Still Evaluating、Too Early、Conservative、Remain Cautious。

### 十、法說會與前次財測比較

若資料中有前次法說內容，整理：

| 項目 | 上次法說 | 本次結果 | 評估 |
| :--- | :--- | :--- | :--- |

判斷：優於財測／符合財測／低於財測。

若沒有資料，請直接說：**「本次資料不足，無法比較前次財測。」**

### 十一、紅黃綠燈評分

| 項目 | 評價 | 原因 |
| :--- | :---: | :--- |
| 營收 | 🟢🟡🔴 | |
| 毛利率 | 🟢🟡🔴 | |
| EPS | 🟢🟡🔴 | |
| 財測 | 🟢🟡🔴 | |
| AI需求 | 🟢🟡🔴 | |
| 接單能見度 | 🟢🟡🔴 | |
| CapEx | 🟢🟡🔴 | |
| 現金流 | 🟢🟡🔴 | |
| 庫存 | 🟢🟡🔴 | |
| 管理層態度 | 🟢🟡🔴 | |
| 整體風險 | 🟢🟡🔴 | |

每項只選一個燈號並填寫原因；資料不足的項目標示「⚪ 資料不足」。

### 十二、股價影響分析（重點）

請回答：

*   **哪三件事情最可能影響股價？**（依重要性排序，例如：AI需求超預期、毛利率改善、財測上修）
*   **哪些資訊可能讓法人上修預估？**（例如：EPS、毛利率、AI營收、AI Server、CapEx、新客戶）
*   **哪些資訊可能讓法人下修預估？**（例如：ASP下降、客戶遞延、AI投資回收慢、匯率、庫存增加）

### 十三、重要引用

每一項重大結論，都盡可能附上：

*   **原文引用（不超過兩句）**
*   出處：簡報章節（頁碼）／Q&A／逐字稿位置（SRT 時間戳，如 `(01:04.700)`）

> [!CAUTION]
> **不要自行杜撰引用內容。**

## 6. 分析原則

請遵守以下原則：

1.  **不可自行捏造數字。**
2.  事實、管理層說法、法人疑慮、模型推論必須明確區分。
3.  數據優先於敘述。
4.  簡報數字優先於逐字稿。
5.  Q&A 內容的重要性高於管理層預先準備的簡報。
6.  若資料不足，請明確說明，不可自行推測。
7.  重點放在可能影響未來股價與基本面的資訊，而非逐字摘要。
8.  特別留意台灣科技產業常見主題，例如 AI 伺服器、ODM/OEM、CoWoS、HBM、ASIC、半導體製程、雲端資料中心、車用電子、Edge AI 等。
9.  Whisper 字幕存在轉錄錯誤的可能，凡數字與專有名詞應優先以簡報交叉驗證。

## 7. 資料品質檢查與問題回報 (Data Quality Check & Issue Reporting)

> [!IMPORTANT]
> 分析過程中若發現資料來源之間的**問題或不一致**，除了在報告中標示外，必須將問題彙整後提交 GitHub Issue 至
> `https://github.com/wenchiehlee-money/InvestorConference/issues`，供上游資料管線（轉錄、OCR、PDF 轉 MD）修正。

### 7.1 應回報的問題類型

| 類型 | 範例 |
| :--- | :--- |
| **字幕轉錄錯誤** | 人名錯誤（同一人在不同檔案拼法不一）、術語錯誤（「法說會」→「法稅會」、「營收」→「營縮」）、年份/數字錯誤（「2026年」→「2020年」）、Whisper 幻覺句（憑空出現與會議無關的語句） |
| **簡報 OCR 缺漏** | PDF 轉 MD 後某頁僅剩標題、數據表或圓餅圖數字完全遺失、YoY 標籤與數值排版錯位無法對應 |
| **跨檔案不一致** | 字幕數字與簡報數字矛盾、GT 與 FIN 版本內容明顯歧異、QA 檔引用的 webcast 時間戳在字幕中不存在或格式無效（如 `46:67`） |
| **檔案缺漏** | 該季度缺 GT 校正字幕、缺中文/英文簡報 MD、缺 QA 檔（僅於影響分析品質時回報） |
| **Metadata 錯誤** | SRT `[METADATA]` 區塊的 Source/Language 與實際內容不符 |

### 7.2 回報流程

0.  **先跑自動檢查**：`python scripts/lint_sources.py {StockID} {Year} q{N} --issue-draft <暫存檔>`，機械性問題（無效時間戳、OCR 空頁、字幕長度異常）自動產出表格草稿；LLM 再補上語意類發現（人名、術語、數字、幻覺句）。
1.  **再查重**：`gh issue list -R wenchiehlee-money/InvestorConference --search "{StockID} {Year} q{N}"`，若已有相同檔案的 issue，改用 `gh issue comment` 補充，不開重複 issue。
2.  **一季一 issue**（或同公司多季合併為一個 issue），標題格式：
    `[data-quality] {StockID} {公司名}: {受影響檔案摘要} ({Year} Q{N})`
3.  **內文格式**：每個問題一列，必須包含 **檔案路徑、位置（SRT 時間戳或簡報頁碼）、問題描述、建議修正**。範例：

    ```markdown
    ## 資料品質問題彙整

    | 檔案 | 位置 | 問題 | 建議修正 |
    |---|---|---|---|
    | `2357/2357_2026_q1_FIN.srt` | (03:11) | 「新南幣」轉錄錯誤 | 新臺幣 |
    | `2357/2357_2025_q4_qa.md` | Page 7 | `webcast 46:67` 時間戳無效（秒數 67） | 確認應為 46:47 或 46:57 |

    _由 skill-conference-digest 分析 {StockID} {Year} Q{N} 時發現。_
    ```
4.  **提交**：`gh issue create -R wenchiehlee-money/InvestorConference --title "..." --body-file <暫存檔>`。
5.  在 digest 報告的 metadata 表格中加註 issue 連結（欄位名：`資料品質 Issue`）。

### 7.3 判斷原則

*   **不確定是否為錯誤時仍應回報**，但標明「待確認」（例如疑似幻覺句、無法辨識的專有名詞）。
*   純業務面的矛盾（如本季實績低於前次財測）屬第十節分析內容，**不是**資料品質問題，不回報。
*   GT 檔案中的殘留錯誤同樣要回報——GT 是人工校正版，錯誤影響更大。

## 8. 輸出規範

### 8.1 報告檔案

*   **輸出路徑**：`InvestorConference/Conference-digest/{StockID}_{Year}_q{N}_digest.md`
*   **報告開頭**必須包含 metadata 表格：

```markdown
# {StockID} {公司名稱} {Year} Q{N} 法說會重點萃取與分析

| 欄位 | 內容 |
|------|------|
| 股票代碼 | {StockID} |
| 季度 | {Year} Q{N} |

| 資料品質 Issue | {issue 連結，無問題則填「無」} |
```
